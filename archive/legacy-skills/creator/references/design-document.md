# 设计文档指南

仅在用户明确要求产出设计文档时读取并使用。根据项目规模裁剪章节，不要为了完整而填充无关内容。

## 推荐结构

```markdown
# <项目名称>设计文档

状态：设计草案 | 已确认设计 | 已废弃
版本：0.1
日期：YYYY-MM-DD
确认日期：未确认 | YYYY-MM-DD
验收基线：未确认 | Git commit / tag / 产物版本
替代版本：无 | <版本>

## 1. 背景与目标
- 要解决的问题
- 目标用户
- 核心价值

## 2. 范围
### 首版范围
### 非目标

## 3. 需求
- REQ-01：明确、可验证的核心需求

## 4. 使用场景
- 主要流程
- 关键边界情况

## 5. 现状与约束
- 现有项目或技术基础
- 运行与部署环境
- 时间、成本、兼容性或安全约束

## 6. 方案设计
- 总体方案
- 组件与职责
- 数据流或调用关系
- 数据模型、接口或关键配置
- 错误处理与恢复策略

## 7. 外部依赖与失败策略
- 外部服务或运行依赖
- 超时、失败、降级与恢复

## 8. 工程完整性与部署约束
- 配置来源与各环境差异
- 可提交仓库的非敏感配置
- 必须由 Secret Store 或 CI 注入的秘密
- 不得绕过的安全控制
- 本地开发、测试替代方案及生产隔离措施
- 平台标准配置文件、构建和部署命令
- 必须保留的人工步骤及原因
- 临时分支、兼容逻辑或特殊判断的保留与删除条件

## 9. 安全与运维
- 认证、授权与敏感信息
- 日志、监控和故障排查
- 部署、升级、迁移与回滚

## 10. 实施计划
- TASK-01：任务、依赖、预期产物、完成条件和验证方式

## 11. 验收标准
- AC-01：对应 REQ-01，可观察、可运行或可测试的条件

## 12. 追踪矩阵
| 需求 | 实施任务 | 验收标准 |
|------|----------|----------|
| REQ-01 | TASK-01 | AC-01 |

## 13. 决策记录
- 讨论过的方案
- 最终选择及理由
- 重新评估条件

## 14. 风险与待确认项

## 15. 变更记录
- 版本、日期、变更内容、原因、确认状态
```

## 裁剪规则

- 小型脚本或单用途工具：重点保留目标、范围、行为、实现方案、任务和验收标准。
- Web 应用：补充页面、用户流程、状态、接口、安全、响应式和浏览器验证。
- API 或服务：补充接口契约、鉴权、错误模型、幂等、限流、可观测性和恢复。
- Docker 项目：补充镜像、服务依赖、网络、卷、环境变量、健康检查、启动顺序和升级回滚。
- 现有仓库改造：补充现状、影响范围、兼容性、迁移策略和不应改动的边界。

## 写作要求

- 使用明确名词，避免“相关功能”“适当处理”等模糊措辞。
- 将需求、设计选择和实施任务分开。
- 为关键任务定义完成条件。
- 每项核心需求至少对应一个实施任务和验收标准。
- 将安全和运维放进设计，不作为结尾的泛泛提醒。
- 将环境差异、平台配置和部署自动化写成明确约束，不留给实现阶段猜测。
- 不编造尚未确认的仓库结构、依赖版本、接口或业务规则。
- 将未知但不阻塞成文的内容列为待确认项。
- 已确认版本发生重大变更时更新版本和变更记录，不静默覆盖原基线。
