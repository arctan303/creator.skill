# 自进化协议

## 目的

自进化系统负责把真实使用中的问题转成对 Skill 的补丁，而不是无限新增规则。

它要解决两件事：

- 补上反复出现的缺口
- 删除或改写长期无用、造成摩擦的规则

## 信号来源

只有真实使用信号才进入自进化：

- 用户纠正
- 用户明确不满意
- Agent 多次误解同类意图
- Skill 漏掉关键验证
- 审查发现本该由前置 Skill 防住的问题
- 某条规则经常造成不必要阻塞
- 某条规则长期没有发挥作用

不要因为理论上“可能更好”就修改 Skill。

## 记录位置

原始信号记录在：

```text
.codex/evolution/signals.md
```

信号只记录问题和上下文，不在执行关键任务时随手改规则。

每条待处理信号必须保留 `状态：待处理`。这就是新话题启动时判断是否需要自进化的触发物。

## 处理时机

自进化默认在下一次新开话题时处理。新话题读取 `AGENTS.md` 后，如果 `.codex/evolution/signals.md` 存在 `状态：待处理`，主 Agent 应静默派发 fresh 子代理处理，不耽误当前主线。

处理时由 `self-evolver`：

1. 读取信号
2. 判断问题属于哪个 Skill
3. 找到对应 Skill 的现有规则
4. 判断是新增、修改、合并还是删除
5. 直接编辑 `.agents/skills/` 下对应 Skill
6. 把信号状态改成 `已转补丁` 或 `已忽略`
7. 记录简短处理说明

## 补丁原则

补丁是编辑，不是堆叠。

每次补丁都要回答：

- 当前哪条规则没覆盖这个问题
- 是补新规则，还是改旧规则
- 有没有旧规则应该删掉或合并
- 修改后是否更短、更准、更容易执行

低风险补丁可以静默应用，例如澄清触发条件、补验收标准、合并重复规则、删除明显无用规则。

以下情况必须先让用户确认：

- 改变系统主流程
- 降低交付或审查标准
- 影响权限、安全、隐私或发布边界
- 跨多个 Skill 大范围改写
- 信号证据不足，只是推测可能更好

## 禁止事项

- 不在执行关键任务中途随手改 Skill
- 不把单次偶发问题升级成永久规则
- 不只新增不删除
- 不把用户偏好伪装成通用规则
- 不让自进化阻塞用户当前主线任务
