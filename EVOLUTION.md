# 自进化引擎协议 (Evolution Engine)

本文件定义系统自我进化的原则与机制。自进化是为了让系统“越用越准”，而不是“规则越写越胖”。

## 自进化层级（四层结构）

系统进化依据信号频率和严重程度，分为四层逐级递进：

- **第一层：经验积累**
  在日常对话中，当遇到用户修正、多次误解、代码反复出错等情况时，静默收集错误信号到 `.codex/evolution/signals.md`（或 `.claude/` 对应目录），状态标记为 `待处理`。
- **第二层：规则毕业（频次触发）**
  当同类问题出现 3 次以上时，必须提议上升为全局规则或修改现有 Skill。
- **第三层：技能降级（评分触发）**
  基于四维评价模型，当某项指标持续低分（≤3分）时，提议对该 Skill 进行彻底重构或打补丁。
- **第四层：模式识别（盲区触发）**
  当发现某类操作反复出现（5次以上）且目前没有合适的 Skill 覆盖时，提议通过 `skill-builder` 创造新 Skill。

## 四维技能评价模型 (Skill Scoring)

`self-evolver` 在处理长期信号时，应对相关 Skill 进行量化打分（1-5分），以决定进化的方向：

1. **精准度 (Accuracy)**：Skill 指引是否准确无误？
   *（5: 零修正 / 4: 微调 / 3: 修正3处以上 / 2: 方向重做 / 1: 失败弃用）*
2. **覆盖度 (Coverage)**：Skill 是否覆盖了实际的业务需要？
   *（5: 完全按指引 / 4: 极少自由发挥 / 3: 较多临时决策 / 2: 大量自由发挥 / 1: 严重不匹配）*
3. **效率 (Efficiency)**：Agent 执行该流程是否顺畅？
   *（5: 一次通过 / 4: 1次澄清 / 3: 2-3次来回 / 2: 多次来回 / 1: 卡死）*
4. **满意度 (Satisfaction)**：用户对产出的接受度？
   *（5: 主动赞赏 / 4: 默默接受 / 3: 提出修改意见 / 2: 要求大幅修改 / 1: 完全否定）*

**反膨胀机制**：
每次处理信号并撰写补丁时：
- 必须分析是哪个维度的分数导致了问题。
- **优先精简和修改已有规则**，而不是在文件末尾新增大段文字。
- 如果规则超过 200 行且评分依然低，提议拆分而不是继续增加内容。

## 执行机制

- 信号收集是日常的：主 Agent 发现不对劲，就往 signals 文件里写一行记录。
- 信号处理是异步的：新开话题时，如果检查到有 `待处理` 信号，静默派发 **fresh 子代理**运行 `self-evolver`，不阻塞主线对话。

处理时由 `self-evolver`：

1. 读取信号
2. 判断问题属于哪个 Skill
3. 找到对应 Skill 的现有规则
4. 判断是新增、修改、合并还是删除
5. 直接编辑 `.agents/skills/` 下对应 Skill
6. 把信号状态改成 `已转补丁` 或 `已忽略`
7. 记录简短处理说明

## 补丁原则

补丁是编辑，不是堆叠。

每次补丁都要回答：

- 当前哪条规则没覆盖这个问题
- 是补新规则，还是改旧规则
- 有没有旧规则应该删掉或合并
- 修改后是否更短、更准、更容易执行

低风险补丁可以静默应用，例如澄清触发条件、补验收标准、合并重复规则、删除明显无用规则。

以下情况必须先让用户确认：

- 改变系统主流程
- 降低交付或审查标准
- 影响权限、安全、隐私或发布边界
- 跨多个 Skill 大范围改写
- 信号证据不足，只是推测可能更好

## 禁止事项

- 不在执行关键任务中途随手改 Skill
- 不把单次偶发问题升级成永久规则
- 不只新增不删除
- 不把用户偏好伪装成通用规则
- 不让自进化阻塞用户当前主线任务
