# 自进化引擎协议 (Evolution Engine)

自进化用于修正规则系统的真实缺陷，不用于记录个人偏好，也不以增加规则数量为目标。完整闭环是：**安装时生成台账 → 事件发生时采集 → 同类信号累计 → 达到阈值后处理 → 验证并回写结果**。

## 信号台账

- Codex：`.codex/evolution/signals.md`
- Claude Code：`.claude/evolution/signals.md`

发布包必须预建对应文件。若旧项目缺少该文件，主 Agent 在首次检查时创建。推荐格式：

```markdown
# 自进化信号

| ID | 关联规则 | 问题类别 | 摘要 | 首次出现 | 最近出现 | 次数 | 严重度 | 状态 |
| --- | --- | --- | --- | --- | --- | ---: | --- | --- |

## 证据

<!--
### EVO-001
- 用户纠正 / 报错 / 测试 / 审查 / 规则冲突：
- 处理说明：
-->
```

状态只使用：

- `观察中`：真实但尚未达到处理阈值。
- `待处理`：达到频次阈值或属于严重事件。
- `待确认`：补丁会改变主流程、边界或多个 Skill，等待用户决定。
- `已转补丁`：已修改并验证规则。
- `已忽略`：已有证据证明不属于系统规则问题；必须写明理由。

## 何时采集

出现以下任一事件时，在当前关键操作结束后的第一个安全检查点记录或合并信号：

1. 用户明确纠正 Agent 对目标、事实、范围或验收标准的理解。
2. 用户因规则未执行而重复同一要求。
3. 测试或审查发现现有规则本应提前阻止的问题。
4. 规则与宿主能力、项目环境或另一条规则冲突，导致流程无法执行。
5. 同类人工绕行或流程摩擦再次出现。
6. 用户明确要求记住教训、以后避免或优化工作流。

以下内容不采集：个人偏好、无证据推测、一次性网络或外部服务故障、已经由项目需求文档管理的产品变更。

## 四层进化机制

### 第一层：观察积累

普通首次信号标为 `观察中`。使用“关联 Skill + 问题类别”作为合并键；同类事件增加次数并追加证据，不能因为只出现一次就标记为 `已忽略`。

### 第二层：规则毕业

同类信号累计达到 3 次时改为 `待处理`，评估修改现有 Skill 或全局协议。只在现有结构无法覆盖时新增规则。

### 第三层：严重事件直达

单次事件若造成或可能造成安全、隐私、数据损坏、错误发布、越权或其他不可逆后果，直接标为 `待处理`，不等待 3 次。

### 第四层：能力盲区

同类操作出现 5 次以上且没有现有 Skill 覆盖时，提议通过 `skill-builder` 创建新 Skill。创建前必须确认它不是现有 Skill 的触发或说明缺陷。

## 技能评价

处理 `待处理` 信号时，可按以下维度评估相关 Skill：

1. **精准度 (Accuracy)**：指引是否导向正确结果。
2. **覆盖度 (Coverage)**：规则是否覆盖真实场景与边界。
3. **效率 (Efficiency)**：流程是否产生无必要的往返或阻塞。
4. **满意度 (Satisfaction)**：用户是否明确表达接受、修改或否定。

评分必须附证据。无法观察的维度记为 `N/A`；尤其不得把用户沉默当作满意。任一有证据维度 ≤3 分时，必须检查现有规则是否需要修改，但低分本身不自动授权高风险改动。

## 补丁原则

按以下顺序选择动作：

1. 修改未生效或含糊的旧规则。
2. 合并重复或冲突规则。
3. 删除明显无效、过时的规则。
4. 只有前三步无法覆盖时才新增。

低风险补丁可以直接应用，包括澄清触发条件、修正路径或状态、补充验收证据、合并重复规则。以下变化必须先让用户确认：

- 改变系统主流程。
- 降低交付或审查标准。
- 影响权限、安全、隐私或发布边界。
- 跨多个 Skill 大范围改写。
- 证据不足但影响较大。

每次补丁都要记录：缺陷证据、修改或删除的规则、关联的 Prompt 行为回归用例、验证方式、最终状态。回归用例应在修改前复现缺陷、修改后验证期望路线、风险、追问、状态或资源边界；无法形成用例时只保留信号，不应用补丁。文件超过 200 行且仍持续低分时，优先精简或拆分，不继续堆叠。

## 执行与隔离

新会话或安全检查点发现 `待处理` 信号时：

- 宿主支持 fresh 子代理：派发 `self-evolver`，提供信号、协议、目标规则和证据，不传入主 Agent 的预设结论。
- 宿主不支持子代理或后台任务：明确披露限制，在主线任务安全检查点由当前会话处理；不得假装已经异步运行。

自进化不得阻塞发布、迁移、写盘等关键操作，也不得在执行关键任务中途顺手改规则。
