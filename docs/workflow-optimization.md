# creator.skill 工作流优化记录

## 1.2.0 决策结论

- 主 Agent 中文名为“创造者”，英文名采用 **Creovator**（Creator + Innovator）。仓库名和 Skill 名继续使用 `creator`，避免破坏兼容性。
- 人格文案必须落到执行动作：拆分表面诉求与真实目的，识别用户、场景、成功证据、约束和非目标，并区分已确认、合理推断和尚待确认。
- 每轮优先追问 1～2 个会改变方向、范围或验收方式的问题；信息足够后停止追问。
- AI 不是所有产品的默认目标，只在价值、成本与错误边界合理时采用。
- reviewer 采用 R0/R1/R2 分级；R2 和产品 Phase 交付要求 fresh reviewer。
- 设计规范不再从审美词映射固定皮肤，而是记录“意图 → 决策 → 参数 → 理由/代价 → 验证检查点”。
- `release-builder` 按仓库事实识别技术生态，不默认 npm，也不在未授权部署时声称完成在线验证。
- 发布包以 `.creator-manifest.json` 声明平台、版本和技能清单；gateway 保留语义版本与已有自进化台账。

## 三路由架构

用户已确认用以下三条路线替代原来的单线流程：

1. **0→1 产品**：新产品、目标用户、价值主张或核心范围未确定。生成 Product Spec；涉及界面时生成 Design Brief；再形成完整 DEV-PLAN。
2. **产品变更**：已有产品新增能力，或改变用户可见行为、公开契约、数据、权限、流程。只维护增量需求、影响分析和增量计划，不重写无关章节。
3. **维护执行**：修复既定行为、测试/CI、内部重构、文档和依赖维护。允许在没有 Product Spec 或 DEV-PLAN 时启动，但执行前必须建立维护任务契约。

维护任务契约包含：目标/预期行为、现象与证据、范围、非目标、完成证据/验证命令、R0/R1/R2 风险级别。

以下情况必须由维护执行升级到产品变更：

- 正确行为无法从现有代码、测试、文档或用户说明中确定。
- 修改公开 API、CLI 行为、用户配置契约、持久化格式、公开数据模型、权限、安全边界或用户流程。
- 修复需要用户在多个合理行为之间做产品选择。
- 原任务表述为“优化/重构”，实际却扩大用户可见范围。

R0/R1 的短契约可以保留在当前会话；R2、跨会话、多阶段或多人协作任务写入 `docs/maintenance/<task-id>.md`。

## 自进化闭环

- 安装时生成信号台账。
- 用户纠正、测试/审查失败、规则冲突和重复人工绕行会在安全检查点被采集。
- 普通信号首次进入“观察中”，同类累计 3 次转“待处理”；安全、隐私、数据损坏、错误发布和越权事件单次直达。
- 补丁优先修改、合并或删除旧规则；只有现有结构无法覆盖时新增规则。
- 改变主流程、审查标准、安全边界或多个 Skill 的补丁必须先确认。

## 已修正的结构缺陷

- Codex 与 Claude Code 发布包都预建对应的 evolution/signals.md。
- 信号台账拥有稳定字段、合并键、严重度和状态；不再把用户沉默视为满意证据。
- 产品文档默认路径统一到 `docs/`，同时兼容旧项目根目录。
- `reviewer` 的 YAML 字段可以通过严格解析。
- 构建器从源码动态发现 Skill，缺少命令映射或界面字段时直接失败。
- 验证器检查 manifest、版本、技能/命令数量、平台残留和 gateway 结构；CI 构建后必须通过验证才创建 Release。
