# 计划技能

把产品和设计文档拆成开发步骤。用户不需要自己死磕怎么拆，拆分本身就是这个技能的职责。

## 核心原则

每个 Phase 都必须能跑、能看或能被验证。不要写出“代码很多，但什么都验收不了”的阶段。

## 工作方式

1. 读取 `Product-Spec.md`，以及可用的 `Design-Brief.md` 和设计产物。
2. 识别依赖、风险和最小可交付切片。
3. 将工作拆成一个个可独立验收的 Phase。
4. 为每个 Phase 定义任务、验收标准、验证命令和交付证据。
5. 把假设、阻塞和需要用户决策的内容从计划任务中分离出来。

## DEV-PLAN.md 必须包含

- Phase 列表、目标和依赖
- 任务编号，例如 `TASK-01`
- 对应需求编号，例如 `REQ-01`
- 对应验收标准，例如 `AC-01`
- 每个 Phase 的运行、测试或检查方式
- 每个 Phase 的预期产物和证据要求
- 风险、阻塞和待确认问题

## 合格标准

开发步骤文档合格的标准是：开发技能可以拿下一个 Phase 直接开工，并知道要做什么、怎么验证、交付什么证据。

## 红线

- 不按纯技术层拆出不可验收阶段。
- 不把验收设计丢给开发阶段临时决定。
- 不把高风险未知项伪装成普通任务。
- 能按用户可见的垂直切片拆分时，不优先按底层模块拆。
