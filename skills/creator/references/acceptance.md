# 验收指南

仅在用户报告实施完成并希望检查成果时读取。

## 建立验收基线

优先使用用户确认的设计版本，并记录实际验收的 commit、tag、版本号或产物标识。若设计已在实施中变更：

1. 识别实际偏离。
2. 询问或查找变更理由。
3. 判断它是合理替代、范围变更还是遗漏。
4. 判断变更是否已被用户确认并成为新基线。
5. 不要机械地以“与文档不同”为由判定失败。

没有设计文档或只有草案时，可以检查质量，但必须说明无法完成严格的设计符合性验收。

读取 `project-context.md` 和 `implementation-progress.md`（如果存在），但以已确认设计为验收依据。进度记录不能替代设计确认。

## 逐项追踪

- 按 `REQ` 编号确认需求是否实现。
- 按 `AC` 编号执行或检查验收条件。
- 引用构建日志、测试结果、截图、响应、文件或运行观察作为证据。
- 对每一项标记：通过、失败、无法验证或已确认偏离。
- 核心需求缺少验收标准时，标记为基线缺陷，不自行编造通过条件。

## 按产物选择证据

### Web 应用

- 构建与测试结果
- 关键页面和主要用户流程
- 空状态、错误状态和加载状态
- 响应式布局与目标浏览器
- 网络请求、鉴权和敏感信息暴露
- 浏览器控制台及服务端错误

### CLI、脚本和开发工具

- 安装或启动方式
- 正常输入、边界输入和非法输入
- 退出码、标准输出和错误输出
- 文件与网络失败场景
- 幂等性或重复执行行为

### API 与后端服务

- 构建、测试与启动
- 接口契约和错误响应
- 认证、授权、输入校验和速率限制
- 数据一致性、并发、重试和幂等
- 日志、健康检查和故障恢复

### Docker 与 Compose

- 镜像能否构建
- 容器能否从干净环境启动
- 服务依赖、网络、卷和环境变量
- 健康检查与重启行为
- 数据持久化
- 升级、回滚和敏感信息处理

### 编译型项目

- 干净环境下编译
- 单元、集成或端到端测试
- 目标平台和构建配置
- 运行产物与错误处理
- 打包、版本和发布信息

## 工程完整性检查

验收软件项目时读取 [engineering-integrity.md](engineering-integrity.md)，并检查：

- 环境相关值是否集中管理，是否存在散落的可变 URL、路径、阈值或部署标识。
- 密钥是否进入源码、前端构建产物、仓库配置或日志。
- 开发模式是否绕过生产必须保持的鉴权、人机验证、限流或授权控制。
- 本地开发替代方案是否明确受控，并且无法进入生产构建。
- 项目是否使用目标平台支持的标准配置、构建和部署方式。
- 非敏感配置是否能从仓库和 CI 重建，是否存在无理由的后台手工录入步骤。
- 临时判断、兼容分支和无效代码是否有真实场景与删除条件。

不要把所有代码常量判定为硬编码问题。只报告会随环境、部署或运营变化却散落在代码中的值，以及本应保密却进入代码或仓库的内容。

## 问题分类

- 阻塞：核心目标或明确验收标准未满足。
- 重要：主要功能可用，但可靠性、安全性或部署存在实际风险。
- 轻微：不阻止交付的局部问题。
- 优化：超出当前验收范围的改进建议。
- 无法验证：缺少环境、凭据、依赖、数据或可访问产物。

## 报告格式

```markdown
# 验收报告

结论：通过 | 有条件通过 | 不通过 | 无法完成验收

## 验证证据
| 编号 | 结果 | 执行或检查内容 | 证据 |
|------|------|----------------|------|
| AC-01 | 通过/失败/无法验证 | ... | ... |

## 必要整改
- 问题、影响、复现或证据、通过条件

## 合理偏离
- 与设计不同但可接受的实现及理由

## 无法验证
- 缺少什么，它影响哪些结论，以及补齐后如何继续

## 后续优化
- 不影响本次验收的建议
```

不得只凭代码看起来正确就判定通过。无法运行某项验证时，明确标记，不得伪造结果。
