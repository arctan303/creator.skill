# 项目调查指南

在评价或设计现有仓库前读取。目标是获取足够可靠的项目背景，不是遍历和总结全部文件。

## 调查顺序

1. 查看仓库根目录、Git 状态和现有文档。
2. 识别语言、框架、依赖、构建和启动方式。
3. 找到与当前需求直接相关的入口、模块、接口和测试。
4. 检查部署、Docker、CI、环境变量示例和迁移配置。
5. 必要时查看相关 Git 历史，理解非显然设计的来源。
6. 停止读取无关文件，开始形成项目地图。

## 优先材料

- `README`、`CONTRIBUTING`、`docs/` 和架构决策记录
- 包管理、锁文件、构建和编译配置
- 应用入口和目标功能模块
- 测试目录及测试命令
- Dockerfile、Compose、CI 和部署配置
- 数据库 schema、迁移、API 定义和公共类型
- 环境变量示例，不读取或复制真实密钥

## 输出心智模型

调查后应能说明：

- 项目解决什么问题，当前有哪些能力
- 与本次设计相关的模块和调用关系
- 如何构建、测试、运行和部署
- 哪些接口、数据或行为需要保持兼容
- 已知限制、技术债和潜在风险
- 哪些结论来自文件，哪些只是暂时假设

不要默认向用户输出完整调查报告。只在讨论需要时引用相关事实。
