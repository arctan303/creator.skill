# 自进化阶段契约

## 输入

- `EVOLUTION.md`
- `.codex/evolution/signals.md` 或 `.claude/evolution/signals.md`
- 相关 Skill、`AGENTS.md` 与可验证证据

## 台账状态

- `观察中`：真实信号，尚未达到处理阈值。
- `待处理`：累计 3 次，或单次属于安全、隐私、数据损坏、错误发布等严重事件。
- `待确认`：拟议改动影响主流程、系统边界或多个 Skill。
- `已转补丁`：补丁已经应用并验证。
- `已忽略`：证据表明不属于规则缺陷；必须保留理由。

## 合并规则

使用“关联 Skill + 问题类别”识别同类信号。重复事件增加次数、更新最近出现日期并追加证据；不得为同一问题反复新建条目。

## 可采集证据

- 用户明确纠正或因规则未执行而重复要求
- 报错、测试失败或审查结论
- 规则之间、规则与宿主能力之间的可复现冲突
- 同类人工绕行或流程摩擦
- 用户明确要求沉淀教训

不得采集没有证据的理论偏好、个人习惯、一次性外部故障，或普通产品需求变更。

## 输出

- 更新后的信号台账
- 被修改、合并或删除的规则及理由
- 验证命令或静态检查结果
- 需要用户确认的高风险变化

## 补丁策略

1. 先修改现有规则。
2. 能合并就合并。
3. 能删除无效规则就删除。
4. 只有现有结构无法覆盖时才新增。

每个已应用补丁必须关联至少一个 Prompt 行为用例：修改前能复现缺陷，修改后能验证期望路线、风险、追问、状态或资源边界。无法构造回归用例时保留信号，不转补丁。

评分只能使用可见证据；未知项记为 `N/A`，不得从用户沉默推断满意度。

## 必须用户确认的变化

- 改变主流程顺序。
- 降低交付、审查、隐私或发布标准。
- 改变权限、安全或真实发布边界。
- 大范围跨多个 Skill 重写。
- 信号证据不足但影响较大。

## Examples

### 首次普通信号

一次模型容量不足属于外部故障，先记录“观察中”；若工作流同时无界生成 reviewer，则把“重复审查放大资源消耗”作为可复现规则缺陷单独记录。

### 单次严重信号

规则导致未经授权自动发布或打印真实密钥时，可直接标为“待处理”，但仍必须补回归用例并验证补丁。

### 非信号

用户只说“我更喜欢另一种标题风格”，且没有失败、纠正或契约冲突证据，不应驱动全局规则进化。
