# 自进化技能 (Self-Evolver)

把真实的纠正、失败和流程摩擦转化为可追溯的规则改进。先积累证据，再修改系统；目标是越用越准，不是规则越写越多。

## [信号文件]

Codex 使用 `.codex/evolution/signals.md`，Claude Code 使用 `.claude/evolution/signals.md`。文件不存在时，按 `EVOLUTION.md` 中的模板创建，不得因为文件缺失而跳过采集。

每条信号至少包含：

- 信号 ID
- 关联 Skill 或 `AGENTS.md`
- 问题类别与摘要
- 首次出现和最近出现日期
- 次数
- 严重度：`普通 / 严重`
- 状态：`观察中 / 待处理 / 待确认 / 已转补丁 / 已忽略`
- 证据：用户纠正、报错、测试、审查结果或规则冲突

## [何时采集]

出现以下任一真实事件时记录信号：

- 用户明确纠正 Agent 对目标、事实、范围或验收标准的理解。
- 用户因为规则未执行而重复同一要求。
- 测试或审查发现了现有规则本应提前阻止的问题。
- 规则与宿主能力、项目环境或其他规则冲突，导致流程不可执行。
- 同类人工绕行或流程摩擦再次发生。
- 用户明确要求沉淀教训或优化工作流。

不要记录纯个人偏好、没有证据的推测或一次性外部服务故障。写信号不得打断发布、迁移、写盘等关键操作；在当前操作完成后的第一个安全检查点补记。

## [采集与晋级]

1. 读取 `EVOLUTION.md` 与现有信号台账。
2. 用“关联 Skill + 问题类别”查找同类信号；找到则增加次数并追加证据，不重复新建。
3. 普通首次信号标为 `观察中`，继续保留，绝不能直接标记为 `已忽略`。
4. 同类信号累计达到 3 次时改为 `待处理`。
5. 单次涉及安全、隐私、数据损坏、错误发布或不可逆操作时，直接标为 `待处理`。
6. 只有证据表明信号不属于系统规则问题时，才标为 `已忽略`，并写清理由。

## [处理工作流]

1. **定位规则**：找到造成问题或未覆盖问题的现有条款。
2. **证据评分**：按精准度、覆盖度、效率、满意度评估相关 Skill。只能依据可见证据评分；用户未表达满意度时记为 `N/A`，不得把沉默视为认可。
3. **选择动作**：依次考虑修改旧规则、合并重复规则、删除无效规则，只有现有结构无法覆盖时才新增。
4. **控制范围**：优先修改一个 Skill；需要跨多个 Skill 或改变主流程时，先把状态改为 `待确认` 并向用户说明影响。
5. **验证补丁**：检查触发条件、文件路径、状态流转和上下游契约是否仍一致；能运行验证脚本时必须运行。
6. **回写台账**：应用完成后标为 `已转补丁`，记录修改文件和验证证据；未应用则保留 `待确认` 或写明 `已忽略` 理由。

## [执行隔离]

宿主支持 fresh 子代理时，使用不继承主会话判断的实例处理 `待处理` 信号，并提供信号、协议、目标规则和必要证据。宿主不支持子代理或后台任务时，在主线任务到达安全检查点后于当前会话处理，并明确披露该限制；不得声称已经异步执行。

## [补丁确认规则]

- **可直接应用**：澄清触发条件、修正路径或状态错误、补充验收证据、合并重复规则、删除明显无效条款。
- **必须先确认**：改变主流程、降低交付或审查标准、改变权限/安全/隐私/发布边界、跨多个 Skill 大范围重写、证据不足但影响较大。

## [红线]

- 不把用户偏好伪装成通用规则。
- 不在关键操作中途修改系统规则。
- 不把首次普通信号直接忽略。
- 不根据用户沉默推断满意度。
- 不只新增不删除；文件超过 200 行时优先精简或拆分。
