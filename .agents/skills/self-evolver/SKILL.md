# 自进化技能 (Self-Evolver)

把真实的报错、纠正和摩擦，转化为定量的技能评估，并打出“越用越准”的补丁。目标是精简优化，不是规则膨胀。

## 工作流：四层进化引擎

1. **读取信号**：读取 `EVOLUTION.md`（协议准则）和 `.codex/evolution/signals.md`（或 `.claude/evolution/signals.md`）中的 `待处理` 信号。
2. **频次与模式分析**：
   - 检查该信号是否属于单次偶发？若是，直接忽略并标记状态。
   - 检查该类信号是否已累计达到 3 次？若是，触发**【第二层：规则毕业】**，准备将其提名为全局规则或 Skill 正式条款。
   - 检查该动作是否属于反复出现但无 Skill 支持的场景？若是，触发**【第四层：盲区触发】**，提议调用 `skill-builder`。
3. **四维量化打分 (Skill Scoring)**：
   若问题指向现有 Skill，必须对其在本次流程中的表现打分（1-5分，满分5分）：
   - **精准度 (Accuracy)**：是否一次性给出了正确的路径？
   - **覆盖度 (Coverage)**：是否覆盖了业务要求，还是让 Agent 疯狂自由发挥了？
   - **效率 (Efficiency)**：过程中来回确认了几次？
   - **满意度 (Satisfaction)**：用户最后是默默接受还是抱怨了？
   
   *判定规则：只要有任一维度 ≤ 3 分，即触发**【第三层：技能降级预警】**。*
4. **撰写补丁 (反膨胀机制)**：
   针对低分维度撰写补丁。
   - **优先修改和删除**：哪条旧规则没生效？删掉它，换成短平快的新规则。
   - **禁止单纯堆叠**：绝对不允许在文件末尾无脑追加长篇大论。
5. **执行修改**：
   直接编辑 `.agents/skills/` 下的对应文件，或 `AGENTS.md`。完成后，将原信号状态改为 `已转补丁` 或 `已忽略`。

## 补丁确认规则

- **可静默应用（低风险）**：澄清触发条件、补充验收证据要求、删除无用死规则。
- **必须先询问用户（高风险）**：涉及系统主流程变更、安全/隐私/发布边界的放宽、跨多个 Skill 的大范围重构。

## 红线（禁止事项）

- 不把用户的个人偏好（如“我喜欢用 Tab 缩进”）伪装成通用的系统级规则。
- 不在执行关键开发任务的中途跑去改规则。
- 绝不“只新增、不删除”。如果一个文件超过了 200 行，你应该考虑的是精简，而不是继续加。
