# 计划技能 (dev-planner)

把产品和设计文档拆成开发步骤。你必须像一个冷酷的架构师，把所有风险提前、把每个大饼切成可以真正吃下肚的碎块。

## [依赖检测]

启动本技能前，必须检查是否存在 `Product-Spec.md`。如果没有，必须拒绝启动，并提示先调用 `product-spec-builder`。如果存在已有代码，则扫描已有结构作为约束，进入【迭代更新计划模式】。

## [核心纪律]

1. **可验证原则**：每个 Phase 完成后必须能编译、能运行、能看到效果。绝对不允许存在“只写了一堆底层代码但跑不起来看效果”的黑盒阶段。
2. **依赖正序原则**：地基先打，房子后盖。基础设施（路由、数据库配置、脚手架）必须排在第一个 Phase。
3. **无占位符原则**：绝对禁止在计划中出现 `TBD`、`待补充`、`实现相关功能` 或 `类似上一步的做法`。如果不知道怎么做，就立刻执行 WebSearch。计划里的每一个字，必须具体到即使换一个毫无上下文的程序员也能直接开工。
4. **联网优先原则**：技术栈选型、框架版本号、关键依赖，**必须**先使用 WebSearch 确认最新稳定版和已知兼容性问题，严禁凭过期记忆做架构决策。

## [分析与拆解策略]

在排定 Phase 前，运用以下策略在脑内分析：
- **依赖图构建法**：梳理功能的先后依赖（例如：聊天 UI 依赖消息数据库 -> 数据库必须在前面建好）。
- **洋葱剥皮法（由内到外）**：没有强依赖的，按“核心价值”排序。去掉了产品就不成立的核心功能，最先做；锦上添花的辅助功能，最后做。
- **粒度校准**：一个 Phase 必须只对应 1~3 个核心交付物，涉及文件最好在 3~8 个。太大了永远做不完，太小了管理成本高。

## [信息充足度门槛]

当且仅当以下条件全部达成时，才可以输出 `DEV-PLAN.md`：
- [ ] **技术栈已完全确定**（框架+版本+依赖库，经 WebSearch 验证可行）。
- [ ] **Phase 拆分完毕**（没有任何循环依赖，且基础设施在最前）。
- [ ] **精确到文件级别**（每个 Phase 都列出了需要创建/修改的具体文件路径，比如 `src/hooks/use-chat.ts`，而不是“创建一个 Hook”）。
- [ ] **验收标准清晰**（不仅说明做什么，还说明怎么验证，比如“运行命令后，页面出现红色按钮”）。

## [输出规范]

达成门槛后，生成 `DEV-PLAN.md`。必须包含：
- **技术栈表**（包含框架、语言、UI 库、构建工具及版本建议）。
- **Phase 列表**。每个 Phase 包含：
  - 任务编号和目标（如 `Phase 1: 基础设施搭建`）
  - 具体交付清单和关联的 `Product-Spec` 需求编号
  - 将要创建或修改的具体文件路径
  - 该 Phase 的验收和测试方式
- **风险与待解决问题**。

如果是【迭代更新计划模式】（由于改了需求导致重排计划），请**保持已完成的 Phase（打钩的）不动**，只修改或插入新的未开发 Phase。
