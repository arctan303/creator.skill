# 开发计划技能 (dev-planner)

计划必须让另一个开发者可以直接执行，也让 reviewer 可以依据需求和证据判断通过或失败。

## [路线守卫]

- 0→1 产品：生成完整计划。
- 产品变更：只更新受影响的未完成范围，生成增量 Phase。
- 维护执行：不强制调用本技能；使用维护任务契约。发现任务实际改变产品契约时，先升级到产品变更。

## [依赖检测]

按项目约定、`docs/Product-Spec.md`、根目录旧文件的顺序查找 Product Spec。缺失或不可验收时停止。存在代码时扫描仓库、脚本、测试和 CI，把真实结构作为约束。

## [核心纪律]

1. **垂直可验收**：每个 Phase 交付可运行、可观察的能力，不按数据库/后端/前端制造长期黑盒。
2. **依赖正序**：先处理真正的阻塞依赖；基础设施只做到当前能力所需，不先造完整地基。
3. **具体无占位**：禁止 `TBD`、`稍后补充`、“实现相关功能”和无法执行的抽象任务。
4. **尊重既有生态**：已有项目默认沿用当前技术栈、锁文件和约定；只有变更确实需要新技术时才选型并查证当前兼容性。
5. **计划不替用户决策**：产品范围、公开行为、安全边界或高风险未知项必须回到用户或 Product Spec。

## [计划步骤]

1. 映射 `REQ-*`、`AC-*` 和设计约束。
2. 建立依赖关系和影响范围。
3. 按可演示、可测试的能力切分 Phase；每个 Phase 通常包含 1~3 个核心交付物。
4. 为每个 Phase 写明范围、非范围、文件或产物、命令、证据和 R2 审查要求。
5. 检查是否存在循环依赖、不可运行阶段或未经确认的产品决策。

## [输出]

先遵循项目约定；默认写入 `docs/DEV-PLAN.md`。

- 0→1：生成完整计划、技术栈事实、Phase、风险和阻塞。
- 产品变更：保留完成且仍有效的 Phase；新增或修改增量 Phase，并映射变更 ID、需求和验收标准。

每个 Phase 必须包含：目标、范围、非范围、任务、预期文件/产物、依赖、验证命令、交付证据和审查要求。
