# 计划技能阶段契约

## 输入

- `Product-Spec.md`
- `Design-Brief.md`，如果有
- 设计图或原型说明，如果有
- 现有仓库事实，如果是在已有项目中规划

## 输出

默认输出文件：`DEV-PLAN.md`。

目标项目已有文档约定时遵循项目约定；没有约定时放在 `docs/DEV-PLAN.md`。

## 最小结构

`DEV-PLAN.md` 至少包含：

- 计划目标
- Phase 列表、目标和依赖
- 每个 Phase 的范围和非范围
- 任务编号，例如 `TASK-01`
- 映射需求编号，例如 `REQ-01`
- 映射验收标准，例如 `AC-01`
- 运行、测试、检查或审查方式
- 预期文件或产物
- 交付证据要求
- 风险、阻塞和待确认问题

## 合格标准

每个 Phase 必须包含：

- 目标
- 范围
- 具体任务
- 预期文件或产物
- 运行、测试或检查方式
- 验收证据
- 审查要求

整个计划必须：

- 按可验收能力切片，而不是只按技术层切片。
- 让下一个 Phase 可以被开发 Agent 直接执行。
- 让 reviewer 能按需求、验收标准和证据独立判断是否通过。
- 明确哪些问题需要用户决策，不能伪装成普通任务。

## Phase 模板

每个 Phase 建议使用以下结构：

```markdown
## Phase N：<目标>

### 目标

### 范围

### 不包含

### 任务

- TASK-01：...

### 映射需求和验收标准

- REQ-01：AC-01, AC-02

### 验证方式

### 交付证据

### 审查要求
```

## 必须修改计划的情况

- Phase 无法运行、测试、检查或审查
- 依赖关系不清楚
- 验收证据缺失
- Phase 大到无法干净完成和审查
- 计划和已知项目约束冲突
- 任务没有映射到需求或验收标准
- 把用户决策、产品范围变化或高风险未知项当成开发任务
- Phase 完成后没有可读文件、可运行结果或可审查证据

## 打回输出

计划不合格时，输出必须包含：

- 哪些 Phase 不可验收
- 缺少的依赖、证据或用户决策
- 建议如何重新切分
- 当前可先执行的最小 Phase，如果存在
