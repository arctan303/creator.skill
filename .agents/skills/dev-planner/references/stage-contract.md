# 开发计划阶段契约

## 输入

- 可验收的 `Product-Spec.md`
- `Design-Brief.md` 或设计产物（如果相关）
- 现有仓库、脚本、测试和 CI 事实
- 0→1 或产品变更路线

## 输出

默认维护 `docs/DEV-PLAN.md`。

- 0→1：完整 Phase 计划。
- 产品变更：映射变更 ID 的增量 Phase，保留仍有效的已完成工作。

## Phase 最小结构

```markdown
## Phase N：<可验收能力>

### 目标
### 范围与非范围
### 依赖
### 任务与预期文件/产物
### 映射需求和验收标准
### 验证命令与交付证据
### R2 审查要求
```

## 合格标准

- 按可运行、可演示、可审查的能力切片。
- 每个 Phase 映射 `REQ-*` / `AC-*` 或产品变更 ID。
- 任务具体到文件、产物或可执行操作，不含占位符。
- 验证命令和证据足以判断通过或失败。
- 既有项目的技术栈与仓库事实一致。
- 用户决策、公开行为变化和高风险未知项没有被伪装成开发任务。

## 必须打回

- 当前任务实际属于维护路线。
- Product Spec 或变更记录不可验收。
- Phase 无法独立运行、验证或审查。
- 依赖和影响范围不清。
- 已完成 Phase 被无理由改写。
- 任务没有映射需求、变更 ID 或验收标准。

打回时说明不可验收范围、缺少的决策或证据，以及可先执行的最小 Phase（如果存在）。
