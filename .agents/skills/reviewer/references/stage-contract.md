# 审查员阶段契约

## 输入

审查员必须拿到足够上下文：

- 审查目标和范围
- 相关 `Product-Spec.md` 需求编号和验收标准
- 相关 `Design-Brief.md` 约束，如果涉及体验或界面
- `DEV-PLAN.md` 中对应 Phase、任务和验收证据要求
- 变更文件或产物清单
- 验证命令和结果
- 实现者说明的偏离、风险和未覆盖项

## 必查项

- 是否满足当前 Phase、任务或交付物范围
- 是否覆盖对应验收标准
- 是否有行为回归
- 是否有缺失测试或缺失验证
- 是否有安全、隐私、发布风险
- 是否存在未经确认的范围扩大、设计偏离或需求变更
- 证据是否可复现，命令和结果是否足以支撑通过结论

## 输出要求

问题优先，结论明确。代码审查时尽量提供文件和行号。

## 判定标准

- 通过：验收标准被证据覆盖，没有阻塞问题，残余风险可接受。
- 有条件通过：主体可接受，但存在必须在进入下一阶段前补齐的非实现阻塞，例如记录状态、补交证据、说明残余风险。
- 不通过：存在漏需求、行为回归、缺失关键验证、安全隐私风险、范围越界或证据不足以证明完成。
- 无法审查：缺少必要上下文、缺少变更内容、缺少验证结果，或审查员不是 fresh mind。

## 输出模板

```markdown
结论：通过 / 有条件通过 / 不通过 / 无法审查

## 问题

- <严重级别>：<问题、证据、影响、建议处理>

## 验收覆盖

- <REQ/AC 或任务>：已覆盖 / 未覆盖 / 无法判断

## 残余风险

- <即使通过仍需知道的风险>
```

## 必须打回的情况

- 实现者没有提供验证命令和结果。
- 验收标准没有映射到证据。
- 审查目标不清楚。
- 审查员参与过当前实现。
- 发现阻塞问题但实现者要求通过。
