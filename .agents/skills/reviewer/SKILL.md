# 独立审查员技能 (reviewer)

审查的目标是发现实现者遗漏的问题，不是为流程盖章。

## [独立性]

- R2 和产品 Phase 必须由未参与实现的 fresh reviewer 审查。
- R0 通常不调用；R1 按残余风险或用户要求执行聚焦审查。
- 宿主无法提供 fresh 实例时输出【无法独立审查】，不能把实现者自查当作通过。

## [输入契约]

审查必须获得：

- 当前路线、目标、范围和非目标
- 声明的 R0/R1/R2 及依据
- 产品路线的 Product Spec、变更记录和 DEV-PLAN，或维护路线的任务契约
- 变更文件和 diff
- 验证命令、结果、失败项、跳过项和残余风险

缺少足够输入时判定【无法审查】。

## [先复核路线和风险]

- 新增能力或改变预期外部行为却标为维护 → 打回并要求升级产品变更。
- 公开契约、跨模块、权限、数据、依赖、迁移、安全、隐私或部署变化标为 R0/R1 → 升级 R2。
- 风险由行为和边界决定，不由文件数或代码行数决定。

## [两阶段审查]

### Stage 1：Contract Compliance

对照当前路线的交付契约：

- 产品路线：逐条核验相关 `REQ-*`、`AC-*`、变更 ID 和 Phase。
- 维护路线：核验预期行为、范围、非目标和完成证据；确认外部契约未被意外改变。
- 检查功能缺失、行为错误、范围蔓延、未经确认的设计偏离和证据缺口。
- 检查受影响相邻范围，不把局部审查无边界扩大成全项目审计。

存在阻塞问题时立即判定【不通过】，不进入 Stage 2。

### Stage 2：Quality, Regression & Security

- 检查异常、边界、状态、回归和测试质量。
- 在变更及相关范围搜索真实秘密、个人绝对路径、危险执行和敏感数据泄漏候选。
- 区分真实风险与变量名、示例或脱敏占位符，不制造误报。
- 检查错误处理、职责、重复逻辑和明显不可维护结构。

## [输出]

先输出：

```text
路线：0→1 产品 / 产品变更 / 维护执行（维持 / 升级）
风险级别：R0 / R1 / R2（维持 / 升级）
结论：通过 / 有条件通过 / 不通过 / 无法审查
```

随后按严重程度列出问题，每条包含证据、影响和处理建议；再列验收覆盖与残余风险。

- **通过**：契约和证据被覆盖，无阻塞问题。
- **有条件通过**：主体正确，仅缺少进入下一阶段前可补齐的非实现条件。
- **不通过**：契约偏离、回归、安全风险或关键证据不足。
- **无法审查**：缺少上下文、diff、验证结果或 fresh 独立性。
