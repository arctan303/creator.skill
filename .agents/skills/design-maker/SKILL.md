# 设计制图技能 (design-maker)

把抽象的文字规范变成可交付的高保真设计稿/线框图。
你的工作不是胡乱在画布上画画，而是严格地构建一套可复用的设计体系。

## [依赖检测]

启动前必须检查：
1. 按项目约定、`docs/`、根目录同名文件的顺序查找 `Product-Spec.md`（缺了它，不知道画什么功能）。
2. 按同样顺序查找 `Design-Brief.md`（缺了它，不知道用什么风格画）。

## [第一性原则]

1. **组件先行 (Component-First)**：先建可复用组件，再用组件拼装页面；重复出现的同类控件不得在各页面独立维护。
2. **状态完备 (State Completeness)**：画页面不能只画一条最顺利的路径。必须覆盖核心组件和页面的关键状态（空状态 Empty State、加载态 Loading、报错态 Error、悬停态 Hover）。
3. **文档驱动**：不要自己发挥文档里没提到的功能。一切功能模块必须能在 `Product-Spec.md` 中找到对应，一切颜色间距必须能在 `Design-Brief.md` 中找到依据。

## [工作流程与校验]

生成设计稿的过程必须按以下顺序进行：

### 第一步：定义全局设计变量
读取 `Design-Brief.md`，并在脑内（或调用的设计工具中）定义基础变量表：
- 颜色表（主色调、背景色、文字主次色、警告/成功色）。
- 字体系统（标题、正文、小字）。
- 圆角和间距规范（例如：小圆角 4px，大组件圆角 12px）。

### 第二步：提取并创建可复用组件
扫描 `Product-Spec.md`，找出重复出现的 UI 块。
例如：列表项卡片、底栏导航、标准的弹窗确认框、输入表单组。优先将这些画成独立组件。

### 第三步：页面拼装与连线
使用第二步的组件，把 `Product-Spec.md` 里的所有用户流程拼装成页面。

### 第四步：完整性校验清单 (Sanity Check)
画完后，必须执行内部审计，如果有一项没覆盖，回去补全：
- [ ] Spec 里提到的每一个有界面的功能，都有对应的设计图吗？
- [ ] 有数据列表的页面，画了“列表为空”的状态图吗？
- [ ] 有网络请求操作的按钮，画了 Loading 态吗？
- [ ] 关键页面的视觉密度、留白、色彩，符合 Brief 中的调性要求吗？

## [输出规范]

如果在无缝设计工具（如 Figma / Pencil MCP）环境下，直接通知用户设计文件已生成/已更新。
如果是生成说明文档供人类设计师参考，输出的结构必须清晰：包含全局变量表、基础组件库定义、核心页面分解、以及状态切换规则说明。
