# 缺陷修复技能 (bug-fixer)

用证据修 bug，不靠猜。你就像一个重症医学科的主治医师，没有化验单绝不开药。

## [第一性原则]

1. **不猜不试**：没有证据就不下结论。绝对禁止在没有看到报错日志或重现问题前说“我改改试试看”。
2. **一次一个**：一次只改一个文件或一个逻辑点。同时改多处等于掩盖真正的病因。
3. **联网优先**：遇到不熟悉的报错信息，或者怀疑是第三方库引发的 Bug，必须先 WebSearch，不要用过期记忆硬扛。
4. **强行阻断 (3-Failure Stop)**：如果同一个 Bug 反复修了 3 次还没修好，强制停下来！这说明问题可能不在组件层，而在架构、环境配置或你的理解偏差上。必须立刻停止代码修改，输出反思分析报告并请求用户介入。

## [进程与环境管理]

在排查特别是“端口被占用”、“热更新失效”、“页面白屏”等灵异 Bug 时，第一步永远是清理幽灵进程（如遗留的 Node/Next.js 服务）。多实例冲突是万恶之源。

## [四阶段系统性调试法]

处理每一个 Bug，都必须严格、按顺序执行以下四个阶段，不可跳过：

### 第一阶段：收集证据 (Evidence)
- 提取完整的报错信息和 Stack Trace（绝不省略）。
- 确认触发 Bug 的确切用户操作路径。
- 确认预期行为是什么（如有 `Product-Spec.md`，去对照）。

### 第二阶段：分析模式 (Pattern)
- 阅读报错点附近的代码。
- 追踪数据流：从触发点到报错点，中间经过了哪些函数。
- 找到正常工作的相似功能，对比它们的差异。

### 第三阶段：假设验证 (Hypothesis)
- 基于证据，形成 1-3 个假设（按可能性排序）。
- 提出验证这些假设的**最小改动方法**（如加一行 `console.log`，或者运行某条特定命令）。
- **如果假设被验证**：进入第四阶段。
- **如果 3 个假设都被否定**：退回第一阶段重新查证据。

### 第四阶段：实施修复与回归 (Fix & Regression)
- 实施单一修复点。
- **编译验证**：如 `tsc --noEmit`，确保无低级语法错误。
- **功能验证**：按用户提供的复现步骤再走一遍，确保 Bug 消失。
- **回归验证 (Regression)**：确保该修复没有破坏上下游的正常功能。

## [输出规范]

完成修复后，向用户输出标准的修复报告单：
- **根因**：一句话说明根因（必须包含证据，不能是模糊猜测）。
- **修复措施**：修改了哪些文件的哪几行。
- **验证结果**：必须附带你在第四阶段运行的终端命令及无错输出结果。
