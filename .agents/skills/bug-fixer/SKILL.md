# 缺陷修复技能 (bug-fixer)

用证据修 Bug，不靠猜，也不把需求变化冒充修复。

## [路线守卫]

Bug 修复默认属于维护路线。开始前形成维护任务契约：预期行为、现象证据、范围/非目标、完成证据和风险级别。

- 预期行为能由 Product Spec、测试、公开接口、历史行为或用户确认确定 → 继续维护。
- 多个来源互相矛盾，或用户要求的是新的正确行为 → 停止修改，升级为产品变更。

## [核心纪律]

1. **证据先行**：没有现象、日志、失败测试、审查证据或可操作复现路径，不宣称根因。
2. **一次验证一个假设**：可以修改多个必要文件，但每轮只验证一个根因假设，并保持补丁为最小完整修复。
3. **当前资料核验**：陌生错误或第三方依赖问题先查官方资料和当前版本事实。
4. **三次失败停机**：同一问题连续 3 次修复尝试失败，停止代码修改，重新审查架构、环境和预期行为，并请求必要输入。
5. **不破坏环境**：只有证据指向进程冲突时才检查和清理进程；先确认进程归属，不终止无关任务。

## [四阶段调试]

### 1. 收集证据

- 记录完整错误、Stack Trace 或失败测试。
- 确认触发路径、环境、版本、预期行为和实际行为。
- 建立或补全维护任务契约，声明 R0/R1/R2。

### 2. 分析模式

- 阅读报错点和调用链，追踪输入、状态和输出。
- 对比正常路径或最近相关变更。
- 区分已证实事实、合理推断和尚未确认项。

### 3. 验证假设

- 提出 1~3 个按可能性排序的假设。
- 用最小诊断命令、测试、日志或可逆改动逐个验证。
- 三个假设都被否定时回到证据阶段，不继续堆补丁。

### 4. 修复与回归

- 实施最小完整修复。
- 重跑原复现或失败测试，并运行受影响范围的回归检查。
- 根据真实影响确定风险；R2 必须 fresh reviewer。

## [输出]

- 当前路线和风险级别
- 已证实根因、证据和仍未确认的风险
- 修改文件和修复机制
- 原问题验证与回归结果
- 审查结论（如适用）
- 是否发现需要升级为产品变更的契约问题
