# Bug 修复阶段契约

## 输入

- bug 描述
- 复现步骤
- 错误日志或测试输出
- 相关需求、计划或审查意见

## 输出

- 根因说明
- 修复变更
- 回归验证证据
- 未解决风险

## 复现要求

修复前应尽量复现问题。复现证据可以是：

- 失败测试
- 错误日志
- 命令输出
- 操作步骤
- 审查意见中的具体文件和行号

不能复现时，必须记录：

- 尝试过的复现步骤
- 已观察到的证据
- 为什么仍能判断修复方向，或为什么必须阻塞

## 根因要求

根因说明必须区分：

- 已证实根因
- 合理推断
- 仍未确认的风险

不得把猜测写成事实。

## 修复要求

- 优先最小修改。
- 不用大重构掩盖小问题。
- 不静默改变已确认需求或公开行为。
- 涉及范围扩大时，必须回到计划或用户确认。

## 回归验证

必须运行能证明问题已修复的检查，例如：

- 原失败测试
- 新增回归测试
- 构建或类型检查
- 手动复现步骤

如果无法运行验证，必须记录阻塞原因和残余风险。

## 必须打回的情况

- 没有 bug 现象、日志、测试失败或可操作复现信息。
- 修复会改变已确认需求。
- 只能靠吞错误隐藏问题。
- 没有任何回归验证证据却要求声明已修复。
