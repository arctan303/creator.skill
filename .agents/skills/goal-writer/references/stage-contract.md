# Goal 编写阶段契约

## 输入

- 当前上下文或目标
- 相关需求、计划、阶段状态或用户指令
- 需要交给 Agent 自主执行的最小动作

## 输出

- 一个可直接发送给 Agent 的 Goal 块
- 必要时附带不应包含在 Goal 内的背景说明

## 模板

```markdown
## Goal
<一个清楚、单一、可完成的目标>

## 完成标准
- <可观察标准>
- <可观察标准>
- <可观察标准>

## 验收方式
- <需要贴出的命令、截图、文件、审查结果或其他证据>

## 执行边界
- 最大重试次数：<有限整数或按步骤定义>
- reviewer 收敛：<同一 diff 不重复；新 diff 最多一次聚焦复审>
- 中断条件：<模型容量、网络、工具、宿主或外部依赖故障>
- 恢复检查点：<已完成、未完成、最后证据、下一步>
- 授权边界：<不得自行执行的外部状态变更>
```

## 合格标准

Goal 必须：

- 目标单一，不把多个互相依赖的大目标塞在一起。
- 完成标准可观察。
- 验收方式能产生证据。
- 范围足够小，Agent 可以自主跑完。
- 明确需要贴出的命令、截图、文件、审查结果或其他证据。

## 必须打回的坏 Goal

- “完善一下”
- “优化一下”
- “全部实现”
- “把项目做好”
- “你自己看着办”
- 使用“体验更好”“更高级”“更稳定”等不可观察标准
- 省略完成标准或验收方式
- 把需求澄清、设计、开发、发布混成一个目标

## 输出检查

输出前必须确认：

- 是否能判断完成或失败。
- 是否需要用户先决策。
- 是否依赖未确认的外部信息。
- 是否需要 reviewer 或验证命令作为验收证据。

## Examples

### 合格 Goal

“完成当前 Phase 的三个验收标准，运行指定测试和构建；同一 diff 只创建一个 reviewer，新 diff 最多一次聚焦复审。模型容量中断时保存检查点并停止，恢复后从未完成测试继续。未经授权不得发布。”

### 不合格 Goal

“不断生成 reviewer 并持续测试，直到所有人都认为绝对没有问题。”它没有资源边界、停止条件或可判定的完成状态，会制造无限循环。
