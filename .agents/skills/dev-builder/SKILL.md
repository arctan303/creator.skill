# 开发技能 (dev-builder)

代码存在不等于完成。先确认路线和交付契约，再实现、验证、审查和交接。

## [路线与依赖]

- **0→1 产品**：需要可验收的 Product Spec 和完整 DEV-PLAN，按 Phase 执行。
- **产品变更**：需要 Product Spec 的变更记录和增量 DEV-PLAN，只实施受影响范围。
- **维护执行**：需要维护任务契约；R0/R1 可记录在当前回复，R2、跨会话或多阶段任务读取 `docs/maintenance/<task-id>.md`。
- 需要诊断未知根因的 Bug 优先调用 `bug-fixer`。

缺少对应契约时停止；维护中发现目标行为需要重新定义时升级为产品变更，不在代码里替用户决策。

## [核心纪律]

1. **影响范围先行**：修改前说明直接影响、相邻影响、公开边界和回滚方式。
2. **尊重现有实现**：优先使用仓库已有 SDK、模式和工具；外部 API 或依赖用法依赖当前版本时先查官方资料。
3. **最小完整修改**：按职责保持代码清晰；不以固定行数机械拆分，也不为未来假设过度抽象。
4. **验证即证据**：运行与改动相关的真实命令；不能用“看起来没问题”代替输出。
5. **风险分级**：R0 做结构检查，R1 做针对性测试与 diff 自查，R2 或任何产品 Phase 交付必须使用 fresh reviewer。

## [执行闭环]

### 1. 确认契约

- 输出当前路线、目标、范围、非目标、验收标准和 R0/R1/R2。
- 产品路线映射 Phase、`REQ-*`、`AC-*` 或变更 ID。
- 维护路线映射任务契约中的预期行为和完成证据。

### 2. 实施

- 只修改契约范围内的文件和行为。
- 发现范围扩大、设计冲突或公开行为变化时暂停并升级路线。
- 不覆盖用户已有的无关改动。

### 3. 本地验证

- 选择仓库真实存在且能证明本次修改的构建、类型、测试、接口、UI 或结构检查。
- R1/R2 运行针对性回归；Phase 交付还要证明该能力可运行或可演示。
- 失败时先定位并修复；无法修复时记录阻塞、已尝试动作和残余风险。

### 4. 风险复核

- **R0**：检查文档结构、链接、格式或生成结果，并证明无运行行为变化。
- **R1**：复查实际 diff，运行最小但足够的相关测试；影响范围无法证明时升级 R2。
- **R2 / 产品 Phase**：向 fresh `reviewer` 提供目标契约、范围、diff、验证结果、偏离和残余风险；不通过则修复、重验、重审。

### 5. 交接

汇报当前路线、风险级别、修改文件、验证命令和结果、审查结论（如适用）、残余风险和下一步。维护任务完成后不要强行引导进入产品设计或计划。

## [反合理化]

- 文件少、代码少不等于低风险。
- 编译通过不等于业务行为正确。
- “顺手优化”不属于当前契约。
- UI 不能完全自动验证时，仍要验证构建、运行、关键状态，并明确人工检查范围。
- 宿主不支持 fresh reviewer 时，必须披露，不能把自查写成独立审查。
