# 开发技能

根据开发步骤文档完成计划内开发范围，并证明它真的完成。

## 核心原则

任务不是因为写了代码就完成。开发 Agent 必须按 `DEV-PLAN.md` 连续推进所有未完成的计划内开发 Phase 或任务，并自行启动 fresh 子代理审查；只有全部计划内开发范围都有验证证据、审查结论和交付记录时，才允许表明项目开发完成。

## 工作方式

1. 读取 `DEV-PLAN.md`、相关需求和设计文档。
2. 识别所有未完成且属于当前开发目标的 Phase 或任务。
3. 按计划顺序推进，除非 `DEV-PLAN.md` 明确允许并行。
4. 只实现当前 Phase 或任务范围内的内容。
5. 运行必要验证。
6. 自行调用 fresh 子代理审查。这个子代理不能是写代码的同一个脑子，也不能由用户手动补启动。
7. 审查不通过就继续修复、验证、复审，直到通过或记录明确阻塞。
8. 当前 Phase 或任务审查通过且交付证据完整后，记录完成状态，再继续下一个计划项。
9. 所有计划内开发 Phase 或任务完成后，才输出项目开发完成结论。
10. 输出变更摘要、验证证据、审查结果和残余风险。

## 交付证据必须包含

- 修改过的文件或产物
- 验证命令和结果
- 覆盖了哪些验收标准
- 子代理审查结果
- 相对计划的偏离和原因
- 每个 Phase 或任务的完成状态、阻塞项和残余风险

## 合格标准

项目开发合格的标准是：

- 所有计划内开发 Phase 或任务都已完成，或阻塞被明确记录
- 每个完成的 Phase 或任务都符合计划范围
- 必要检查通过
- 每个完成的 Phase 或任务都通过 fresh 子代理审查
- 所有交付证据完整，且开发 Agent 已明确说明项目开发是否可以结束
- 用户可以根据证据判断完成情况

## 红线

- 不静默修改已确认需求、设计或计划。
- 不跳过验证。
- 不自审冒充独立审查。
- 不把代码写完或本地检查通过当成最终完成。
- 不把单个 Phase 完成当成整个项目开发完成。
- 不把启动审查的责任交给用户。
- 不扩大范围来伪装完成度。
- 不隐藏失败检查或跳过的测试。
