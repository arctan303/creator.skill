# 设计规范技能 (design-brief-builder)

像有经验的产品设计师一样访谈：避免把空白问卷丢给用户，优先提供有依据的候选方向。把主观的“感觉”翻译成可实现、可验证的设计决策。

## [依赖检测]

启动本技能前，按项目约定、`docs/Product-Spec.md`、根目录 `Product-Spec.md` 的顺序查找需求文档。如果没有，必须拒绝启动，并引导用户先调用 `product-spec-builder`。你不能在连“产品干什么的”都不知道的情况下谈设计。

只在 0→1 产品或产品变更需要新增/改变视觉规则时使用。维护路线恢复既有视觉行为时读取现有 Design Brief 或设计产物，不重新定义风格；发现视觉“修复”实际改变用户体验规则时升级为产品变更。

## [核心纪律]

1. **有限选项优先**：不要把空白问卷丢给用户。基于产品目标提供 2~3 个有明确差异、理由和代价的方向。
2. **参考拆解法则**：参考产品只用于拆解信息层级、导航、密度、色彩、字体、组件和动效；必须问清用户喜欢或排斥哪一点，不照抄整套皮肤。
3. **场景优先**：设计决策先服务核心任务、内容类型、使用环境和目标用户，再考虑趋势。只有结论依赖当前趋势或组件现状时才联网确认。
4. **可访问性底线**：颜色不能是唯一状态信号；必须考虑对比度、字号、键盘/触控、动效克制和关键状态可辨识性。
5. **拒绝空泛**：“高级感”“科技感”“现代感”只能作为访谈起点，最终必须变成带理由和验证方式的具体变量。

## [审美词转译流程]

收到空泛审美词时，禁止直接套用固定视觉配方。按以下顺序转译：

1. 追问这个词在当前产品中代表什么行为或感受，以及绝对不能像什么。
2. 结合核心任务、内容密度、使用时长、设备、用户能力和品牌约束，找出真正需要解决的设计问题。
3. 给出 2~3 个候选方向；每个方向说明适用理由、代价，并落到层级、密度、色彩、字体、形状、材质、动效和状态反馈。
4. 让用户选择或修正后，写成“设计意图 → 决策 → 参数 → 验证检查点”。

例如，“科技感”可能是高密度监控界面的清晰数据层级，也可能是面向消费者的轻量动态反馈；它不自动等于深色、蓝紫、玻璃态或发光边框。

## [采访维度清单]

必须通过对话集齐以下 4 类决策；已有需求或品牌材料能回答的，不重复询问：

- [ ] **任务与环境**：核心任务、使用频率、设备、光线、单次使用时长和无障碍约束。
- [ ] **信息与交互**：内容层级、密度、导航方式、核心组件及空/加载/错误/禁用状态。
- [ ] **品牌与感受**：希望建立和避免的印象，以及参考或反例中具体喜欢/排斥的元素。
- [ ] **视觉与实现边界**：色彩、字体、形状、动效、现有设计系统和技术限制。

**五秒测试法与矛盾检测**：
如果用户说“我要极致简洁”，但需求文档里一屏要放 20 个图表，应明确指出冲突：“极致简洁和当前图表数量有冲突。可以用折叠面板降低首屏密度，也可以全量展示以优先效率；这个选择会改变信息获取速度和页面复杂度。” 在冲突解决前不擅自决定。

## [输出规范]

达成【采访维度清单】后，按项目已有约定输出；没有约定时生成 `docs/Design-Brief.md`。必须包含：
- **设计目标与锚点**：最终选定的情绪关键词和对照的真实产品。
- **决策矩阵**：每个设计意图对应的具体决策、参数、理由、代价和验证检查点。
- **具体的视觉变量**：颜色与状态、字体层级、形状、材质、间距密度、布局、动效和响应式规则。
- **可访问性约束**：对比度、非颜色状态提示、键盘/触控、字号与动效要求。
- **放弃的方向**：明确记录“不要什么风格”（比如：绝对不要苹果的过度留白风），这对后续开发非常重要。
- **参考与反例**：只记录被借鉴或拒绝的具体要素，不要求照搬完整产品风格。
