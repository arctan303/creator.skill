# 设计规范技能阶段契约

## 输入

- `Product-Spec.md`
- 0→1 产品或产品变更路线
- 用户的风格要求
- 用户提供的截图、品牌、竞品、参考产品或反例

## 输出

目标项目已有文档约定时遵循项目约定；没有约定时输出 `docs/Design-Brief.md`。兼容根目录旧文件，但不同时创建副本。

## 最小结构

`Design-Brief.md` 至少包含：

- 设计目标和目标用户
- 使用环境和关键约束
- 已选视觉方向
- 被放弃的方向和原因
- 调性关键词及其具体解释
- 色彩、字体、间距、密度、布局和组件规则
- 每项关键规则的设计意图、理由、代价和验证检查点
- 对比度、非颜色状态提示、键盘/触控、字号和动效等可访问性约束
- 关键状态和响应式约束
- 用户提供的参考或反例
- 可观察的设计验收检查点

## 合格标准

`Design-Brief.md` 必须：

- 把主观审美转成具体设计决策
- 说明所选方向为什么适合产品
- 记录重要的被放弃方向
- 足以指导设计图或前端实现
- 包含可观察的设计验收检查点

## 主观词转换要求

遇到空泛审美词时，必须先结合任务、用户、内容、环境和品牌语境解释其含义，再转成可执行变量。不得把形容词直接绑定到固定配色、玻璃态、圆角、留白或字体风格。

示例：

- “高级感”必须落到明暗、留白、材质、字体、对比度、动效克制度等选择。
- “简洁”必须落到信息密度、层级数量、组件数量、默认状态和隐藏策略。
- “现代”必须落到布局节奏、字体气质、组件形态、动效和色彩策略。

如果存在多个合理方向，先给用户少量有差异的方向选择，再写入最终规范。

每个最终方向必须写成：`设计意图 → 决策 → 参数 → 理由/代价 → 验证检查点`。参考产品只拆解具体要素，不作为整套视觉皮肤。

## 退出条件

满足以下条件时，才可以进入设计图或开发计划：

- 视觉方向已被选择或有明确临时假设。
- 关键审美词已转成具体设计变量。
- 关键设计决策具有场景理由、代价和可观察检查点。
- 核心状态和可访问性约束已覆盖。
- 设计规则不扩大 `Product-Spec.md` 的首版范围。
- 后续设计或前端实现有可观察检查点。

## 必须继续采访的情况

- 用户只给出空泛审美词
- 同时要求多个互相冲突的风格
- 目标用户或使用环境缺失
- 设计决策会影响产品范围
- 用户给出参考图但没有说明喜欢或排斥的具体点
- 所选视觉方向无法支撑产品目标或使用场景
- 当前请求被标为维护，但实际需要重新定义视觉或交互规则

## 打回输出

继续采访时不要抛长问卷。输出必须包含：

- 已知产品和设计事实
- 当前无法落地的审美词或冲突
- 2 到 3 个可选择方向
- 当前最需要用户确认的问题
